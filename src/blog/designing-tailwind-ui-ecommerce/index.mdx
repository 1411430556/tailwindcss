嗨！我们已经为这个新的 Tailwind UI 电子商务工具包工作了几个月，现在终于接近完成，所以我想写一些关于这个过程的内容，并给你们一个关于我们进展的更新。

设计和构建这样一个大型组件工具包是一个 _巨大的_ 工作。如果你没有关注 Steve 和我开始制作 Tailwind UI 的过程，我们花了大约 10 个月的时间才有了我们觉得值得发布的东西。我们设计了数十个组件，却花了三次的时间将其全部废弃并从头开始。_哇。_

从那时起，我们学到了很多东西，并真正改善了我们的过程，这使得整理这些电子商务创意变得顺利得多。

---

**第一步是研究和分类**。我们开始研究尽可能多的电子商务网站，并对每个我们能辨别出的用户界面模式进行分类，例如：

- 产品概述
- 产品列表
- 产品特性详情
- 类别预览
- 结账表单
- 购物车
- 类别过滤
- 客户评价
- 订单历史
- 订单详情
- 类别大菜单
- 产品快速查看
- 促销部分

在这项研究中，我们详细分析了不同类型的商店，包括像 [Everlane](https://www.everlane.com/) 这样的服装商店，像 [Allbirds](https://www.allbirds.com/) 这样的鞋店，像 [Grovemade](https://grovemade.com/) 这样的办公配件制造商，像 [Casper](https://casper.com/) 这样的床垫公司，像 [Toontrack](https://www.toontrack.com/) 这样的音频插件开发者等等。我们尽量不专注于任何一种特定的 _类型_ 商店，我认为这确实帮助我们识别出哪些模式对于这些类型的网站是普遍适用的，以及不同的想法如何融入这些模式中。

其中一些模式很难识别，尤其是因为电子商务网站与更像宣传册风格的营销网站之间有相当大的重叠。

例如，这个部分其实跟您可能在营销网站上看到的英雄部分并没有太大不同：

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/adidas-hero.jpg)

那么电子商务工具包是否也应该有“英雄部分”，就像营销工具包一样？如果是的话，它们有什么不同——为什么要有两个重复的类别？

经过大量研究和思考，我们最终注意到在电子商务网站上，“英雄”几乎 _从来不是_ 一个真正的英雄。它从来不是像“为现代知识工作者提供的手工制作的办公配件”这样的整体品牌的总结，而总是一些及时的促销信息，比如“夏季末前享受 50% 的折扣”或“我们的秋季系列刚刚推出，现在就来探索”。

我们还注意到这些“英雄”部分通常几乎总是可以非常轻易地移动到页面中间的设计——它们并不是真正的“页眉”内容。这时我们想到了“促销部分”的概念，类似于一个推广销售或新产品的大横幅，_可以_ 作为英雄，但也可以与其他页面内容混合使用。

这种看似微不足道的“小发现”时刻数不胜数，当你看到最终结果时似乎不怎么重要，但要搞清楚这些事情真的是一项艰巨的工作。整理和分类所有这些想法几乎与设计它们一样费力。

---

**接下来是设计完整页面示例**。如果我们从最初的应用 UI 和营销包工作中学到了一件事，那就是在完整页面的上下文中设计的组件会好得多。

在第一组 Tailwind UI 的项目中，我们进行了大量孤立设计，比如仅仅考虑一个不错的分页设计。但是我们发现当我们开始将所有这些东西组合成实际的用户界面时，许多小细节都感觉不太对劲。

有时是字体大小相比于设计中的其他元素显得过大，也许没有使用足够的空白，或者给某个元素提供了过多的对比度，使它与页面的其他部分相对突出，而它本应感觉像是次要内容。当我们重新调整我们的方式，总是从更大的示例中 _提取_ 较小的示例时，一切都开始更协调。

因此，对于电子商务包，我们从一开始就将 _所有内容_ 设计为页面示例。到目前为止，我们设计了大约 50 个完整的页面，包括主页、类别列表、产品页面、结账表单等。

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/50-pages.jpg)

我们会在页面之间重用一些东西以快速推进（如页脚或导航栏的想法），但通常我们会尽量使每个页面的每个核心元素都是独特的，以生成尽可能多的创意。

---

**然后我们构建页面。** 很多人认为我们有一个非常干净的设计交接流程，我们创建这些坚不可摧、完美的 Figma 设计，然后将它们如实地实现成完美的形式，但这完全不是事实。

实际上发生的事情是，我们会将那些 _相当不错_ 的完整页面设计尽最佳努力构建出来，然后在浏览器中一起审查完成的设计以进行调整。通常会发生许多小变化，尤其是一些与间距细节相关的内容，这在浏览器中更容易判断，而不是在设计文件中。然而，有时我们会进行非常大的更改，比如如果我们认为设计看起来不如我们希望的那么好，或者如果准确匹配设计会使代码复杂得多，但与略有不同的设计相比几乎没有好处，就会完全用其他内容替代部分设计。

**接下来，我们提取了各个组件。** 一旦页面设计最终确定并构建完成，我们就将各个元素提取到自己的模板中。这是“产品概述”从产品页面提取、“产品列表”从类别页面提取，以及“促销部分”从商店主页提取的地方。

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/exploded-view.jpg)

这个过程的难点在于，它并不仅仅是“一个页面由组件组成”那么简单——正如你可能知道的，甚至 _组件_ 也是由组件组成的，因此确切决定如何切分 Tailwind UI 的内容始终很具挑战性。

一般来说，我们倾向于选择较大的组件，因为如果有人喜欢某个较大的组件并想在其他地方使用，它很容易提取出一个小部分，但如果你只有小的独立部分，自己想出较大组件的创意并不总是那么简单。

我喜欢将其想象成我们在试图给你提供预构建的 LEGO 创作，而不仅仅是砖块。

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/lego.jpeg)

如果你只需要轮子，拆下卡车的轮子是很容易的——但从头开始将轮子、座位和前灯转变为一个漂亮的卡车就比较难了。

例如，这个单一的“产品概述”部分可以拆分成许多较小的组件：

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/product-overview.jpg)

至少包括一个图像画廊、一个星级评价小部件、一个颜色样本选择器和一个特性手风琴。但我认为以这种整体方式提供它是更好的起点——如果你需要，你仍然可以获取所有这些内容，但作为一个较大的组件，它也充当了 _灵感_，这是使 Tailwind UI 成为一个很酷的产品的重要成分。

我们可能会在未来将许多这些东西提取为较小的组件，但我绝对认为这种“外部向内”的方法总体上是正确的。

---

**最后，进行库存盘点，并重复整个过程。** 我在产品设计中反复学习到的一个教训是，没有多少计划会使第一次尝试的产品正确。你必须不断迭代和改进——这不是一个线性的过程。

对于这个电子商务工具包，我们在设计和构建了一堆页面，提取出所有这些组件后，才能真正看到产品中的漏洞。

我们注意到一开始有许多我们视而不见的东西，例如，我们只有两个大菜单的想法——这可能是因为在设计新页面时很容易不去设计任何导航栏的“打开”状态。所以我们回到设计步骤，想出了许多新概念来丰富类别。

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/mega-menu-concepts.jpg)

我们还注意到，我们在乳白色背景上的卡片式设计数量过多，而简单的平面设计明显不足，我认为这在电子商务设计中更为常见。因此，我们回到最初，设计了一堆新页面以填补这一空白：

![](https://tailwindui.nyc3.cdn.digitaloceanspaces.com/20210806-ecommerce-update/white-backgrounds.jpg)

距离我们希望发布这个工具包还有不到一周的时间，每天我们 _仍然_ 在设计新的内容，以填补我们在接近完成时识别的空缺。这个产品每天都在变得更加出色，我认为这是我们为 Tailwind UI 想出的最佳设计 _(感谢上帝，变得更糟糕将是令人沮丧的。)_
